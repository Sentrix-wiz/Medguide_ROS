WHAT IS THIS PROJECT?
---------------------
MedGuide-ROS is a ROS2-based autonomous hospital delivery robot built on
the TurtleBot3 Burger platform. It navigates a simulated hospital floor,
delivering supplies between rooms using Nav2 path planning, LIDAR-based
obstacle detection, and a mission scheduler with state machine logic.
Everything runs in Gazebo simulation with RViz visualization.


WHAT HAS BEEN BUILT (Phase by Phase)
=====================================

PHASE 1 — ROS2 Foundations
  Files: heartbeat_publisher.py, sensor_echo_node.py, teleop_safety_node.py
  - Heartbeat publisher at 1 Hz for system liveness monitoring
  - Sensor echo node to log raw LIDAR and odometry data
  - Teleop safety node with velocity limits (0.5 m/s linear, 1.5 rad/s angular)
  - 4 unit tests for heartbeat and teleop safety

PHASE 2 — Gazebo Simulation
  Files: gazebo_sim.launch.py, hospital_floor.world
  - Custom hospital floor world (7m x 6m) with 4 rooms
  - Interior walls split with doorways for robot navigation
  - Furniture obstacles: table, chair, bed, cabinet
  - TurtleBot3 Burger robot spawns at dock position (1.0, 1.2)

PHASE 3 — SLAM Mapping
  Files: slam_mapping.launch.py, maps/hospital_map.yaml, maps/hospital_map.pgm
  - SLAM Toolbox (async) for occupancy grid mapping
  - Generated map: 121 x 71 cells at 0.05m resolution
  - Map includes all walls, doorways, and obstacle positions
  - Used for Nav2 localization and path planning

PHASE 4 — Nav2 Autonomous Navigation
  Files: nav2_params.yaml, nav2_navigation.launch.py
  - AMCL localization with auto initial pose at dock (1.0, 1.2)
  - NavFn global planner (A* search algorithm)
  - DWB local controller (dynamic window approach)
  - 2D costmap with static, obstacle, and inflation layers (0.55m radius)
  - Behavior server: spin recovery, backup, drive_on_heading, wait

PHASE 5 — Safety / Obstacle Detection
  Files: obstacle_detector_node.py
  - LIDAR-based obstacle detection in 60-degree forward cone
  - Emergency stop triggers at 0.25m distance
  - Hysteresis clearing at 0.35m to prevent oscillation
  - Publishes /estop topic and overrides /cmd_vel
  - 8 unit tests for thresholds, hysteresis, oscillation prevention

PHASE 6 — Mission Scheduling
  Files: mission_scheduler_node.py, robot_params.yaml
  - State machine: IDLE → NAVIGATING → COMPLETED/FAILED/ABORTED
  - Room-to-room delivery route: Room A → Room B → Room C → Dock
  - Nav2 action client (NavigateToPose) for goal execution
  - Odometry-based distance tracking per goal and per mission
  - Battery simulation with drain rate and abort threshold
  - Services: /start_mission, /abort_mission
  - 11 unit tests for state machine, distance tracking, goal logic

PHASE 7 — Diagnostics and Logging
  Files: diagnostics_node.py, mission_logger_node.py
  - System health: uptime, scan Hz, odom Hz, e-stop state, node count
  - Mission logger writes per-goal results to CSV files
  - Custom ROS2 messages:
    * MissionStatus.msg — mission_id, state, goals, distance, battery, e-stops
    * GoalResult.msg — goal_name, success, duration, distance, straight_line_m

PHASE 8 — Research Infrastructure
  Files: run_experiment.py, analyze_results.py, generate_demo_data.py
  - Experiment runner: N trials with 10s AMCL convergence wait
  - CSV logging: trial, mission_id, goal_name, success, duration, distance
  - Analysis script: pandas statistics + matplotlib comparison charts
  - Demo data generator: 4 conditions x 10 trials = 160 data points
  - Experiment plan document with full methodology

PHASE 9 — Professor-Ready Control System
  Files: medguide_control.py, dashboard.py, run.sh, run_teleop.sh,
         run_mission.sh, run_experiment.sh, setup.sh, test.sh
  - Terminal control panel (medguide_control.py) with interactive menu
  - PyQt5 live dashboard (dashboard.py) with:
    * Dark theme, research-lab style UI
    * Live mission metrics from ROS2 topics
    * Control buttons: Launch Stack, Start Mission, Abort, Teleop, Experiment
    * Experiment results table with path efficiency percentage
    * Event log with timestamps
  - One-command launcher scripts for everything


BUGS FOUND AND FIXED
=====================

1. Nav2 Plugin Names (CRITICAL)
   Problem: Plugin class names used C++ "::" notation instead of ROS2 "/" notation
   Fix: Changed "nav2_navfn_planner::NavfnPlanner" to "nav2_navfn_planner/NavfnPlanner"
   Affected: nav2_params.yaml (5 plugins)

2. Robot Spawn Position (CRITICAL)
   Problem: Robot spawned at (0.5, 0.5) which was inside the costmap inflation zone
   Fix: Moved spawn to (1.0, 1.2) — well inside free space
   Affected: gazebo_sim.launch.py, nav2_params.yaml, robot_params.yaml

3. Solid Walls — No Doorways (CRITICAL)
   Problem: Interior walls were solid with no gaps — robot could not navigate between rooms
   Fix: Split wall segments to create 1.2m doorway gaps
   Affected: hospital_floor.world

4. Timer Leak in Mission Scheduler
   Problem: _finish_goal() created a new timer every call without canceling the old one
   Fix: Single cancelable timer with re-entry guard
   Affected: mission_scheduler_node.py

5. Map Not Loaded
   Problem: yaml_filename in nav2_params.yaml was empty
   Fix: Set absolute path to hospital_map.yaml
   Affected: nav2_params.yaml

6. AMCL Not Converged
   Problem: Navigation goals sent before AMCL had time to localize the robot
   Fix: Added 10-second convergence wait at experiment start
   Affected: run_experiment.py

7. Dashboard Wrong Path (CRITICAL)
   Problem: WS variable resolved to scripts/ directory instead of workspace root
   Fix: Walk 3 levels up from scripts/ to find workspace root
   Affected: dashboard.py


PATH EFFICIENCY METRIC (NEW)
=============================
Formula: efficiency = straight_line_distance / actual_odom_distance * 100%

- straight_line_distance = Euclidean distance from start position to goal
- actual_odom_distance = accumulated from /odom position deltas
- Perfect direct path = 100%, detour path < 100%

Added "straight_line_m" field to GoalResult.msg
Computed in mission_scheduler_node.py for each goal

HOW TO RUN EVERYTHING
======================

Option A — Dashboard (recommended for professor):
  source /opt/ros/humble/setup.bash
  source ~/medguide_ws/install/setup.bash
  python3 ~/medguide_ws/src/medguide_robot/scripts/dashboard.py
  → Click "Launch Stack" → wait 20s → Click "Start Mission"

Option B — Terminal control panel:
  cd ~/medguide_ws
  python3 medguide_control.py
  → Press 1 to launch → Press 2 for WASD drive → Press 3 for mission

Option C — Manual commands:
  Terminal 1: cd ~/medguide_ws && ./run.sh
  Terminal 2: source run_teleop.sh          (WASD drive)
         OR:  source run_mission.sh         (autonomous mission)
         OR:  source run_experiment.sh      (experiment trials)
